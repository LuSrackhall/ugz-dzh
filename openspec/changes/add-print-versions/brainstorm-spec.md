# 手工账电子化系统 — 打印版本设计

## Context

### 背景
项目是一个手工账电子化生成系统，使用 Go + excelize (v2) 生成 Excel (.xlsx) 账本。当前系统生成单一版本的 Excel 文件，金额使用普通数字格式 `#,##0.00` 展示。

### 当前状态
- **技术栈**：Go 1.21+、excelize/v2、标准库
- **输出格式**：单一 Excel 文件（`.xlsx`）
- **金额格式**：普通数字格式 `#,##0.00`
- **账页结构**：
  - 总分类账：A-G 列（日期/凭证号/摘要/借方/贷方/方向/余额）
  - 多科目明细账：A-U 列（A-G 基础 + H-U 明细科目）

### 约束
1. **零手改 xlsx 期初表**：xlsx 中的期初表只是生成品，可由脚本自动重建
2. **每月 xlsx 完整独立**：每月生成时完整复制上月 xlsx 再追加当月数据
3. **科目自动管理**：凭证中新出现的科目自动加入 JSON 配置
4. **凭证解析不改逻辑**：完全保留现有的 HTML 表格解析代码
5. **打印友好**：每月仅标记有变化的账页为"需打印"

---

## Goals / Non-Goals

### Goals
1. **美观打印**：提供金额分栏展示（十亿千百十万千百十元角分），满足传统手工账的美观需求
2. **三套输出**：同时生成查看版 Excel、打印版 Excel、HTML 打印版
3. **数据一致性**：三套输出共用数据源，只需验证查看版即可确认打印版正确性
4. **样式对齐**：三个版本的非金额部分格式保持一致
5. **正反面打印**：HTML 版支持左右对开的正反面打印布局

### Non-Goals
1. **不修改查看版的计算功能**：查看版保持现有数字格式，支持 Excel 原生计算
2. **不引入新的外部依赖**：HTML 版使用 Go 标准库 `html/template`
3. **不实现自动双面打印**：双面打印依赖打印机驱动，系统只提供布局支持
4. **不支持 Web 服务**：这是 CLI 项目，HTML 文件由用户手动打开打印

---

## Decisions

### 决策 1：三套输出方案

**选择**：同时生成三套输出（查看版 Excel + 打印版 Excel + HTML 版）

**理由**：
- 满足不同场景需求：计算验证、快速预览、精美打印
- 查看版用于手工验证，打印版用于快速预览，HTML 版用于正式打印
- 三套输出共用数据源，确保一致性

**替代方案**：
- 双 Excel 输出：缺少 HTML 版的精美打印能力
- Excel + HTML 双版本：缺少打印版 Excel 的快速预览能力

---

### 决策 2：分栏范围

**选择**：所有金额列都分栏（借方、贷方、余额、明细科目）

**理由**：
- 视觉一致性好（不会混用两种格式）
- 满足传统手工账的美观需求
- 符合用户原始需求（"金额的栏使用十亿千百十万千百十元角分的竖格"）

**替代方案**：
- 只对余额列分栏：列数少，易打印，但美观度降低
- 混合方案：根据纸张宽度动态决定，但实现复杂

**代价**：
- 列数增加（总分类账从 7 列 → 约 38 列）
- 可能需要缩小字号（8-9pt）或使用 A3 纸张

---

### 决策 3：样式对齐策略

**选择**：CSS 先行探索样式，确定后落地到 Excel

**理由**：
- 开发阶段高频变更，CSS 修改快（秒级重新生成）
- 浏览器可实时预览效果，调试效率高
- 确定样式后，通过共享的 `TableStyles` 结构体落地到 Excel

**替代方案**：
- Excel 先行：调试效率低，需重新生成 Excel 文件
- 独立维护：三个版本样式不一致，维护成本高

**实现机制**：
```go
// 共享样式定义
type TableStyles struct {
    HeaderBgColor   string
    HeaderFontBold  bool
    DataFontSize    float64
    BorderColor     string
    BorderWidth     int
}

// 应用到 Excel
func (s *TableStyles) ApplyToExcel(f *excelize.File, sheet string) { ... }

// 应用到 CSS
func (s *TableStyles) ToCSS() string { ... }
```

---

### 决策 4：数据一致性保障

**选择**：共用数据源 + 单元测试覆盖 `centsToDigits`

**理由**：
- 三套输出共用同一份 `entries` 和 `initials`，数据层完全一致
- `centsToDigits` 是纯函数，可通过单元测试独立覆盖
- 验证策略：只需验证查看版，即可确认打印版正确性

**保障机制**：
1. `centsToDigits(c int64) [12]int` 纯函数：将金额拆为 12 位数字
2. 独立单元测试覆盖边界值（0、9999999999、-1）
3. 首次生成后做样板打印测试，后续每月只需验证查看版

---

### 决策 5：HTML 版的技术选型

**选择**：Go `html/template` + 单文件 HTML（CSS 嵌入）

**理由**：
- 零新依赖，使用 Go 标准库
- 单文件 HTML 便于分享和归档
- CSS 嵌入确保样式完整性

**替代方案**：
- 外链 CSS：修改样式无需重新生成 HTML，但需要保持文件相对路径
- PDF 生成：需要引入外部依赖（如 chromedp、wkhtmltopdf）

**实现细节**：
- CSS `@media print` 控制打印样式
- CSS `@page :left`/`:right` 控制奇偶页边距
- 模板使用 `embed.FS` 嵌入到二进制文件

---

### 决策 6：CLI 命令扩展

**选择**：扩展 `ledger generate` 命令，支持 `-view-only`、`-print-only`、`-html-only` 参数

**理由**：
- 保持现有命令结构，用户无需学习新命令
- 支持按需生成，减少不必要的输出
- 默认生成所有版本，满足大多数场景

**命令示例**：
```bash
# 生成所有版本（默认）
$ ledger generate -month 2026-01

# 仅生成查看版
$ ledger generate -month 2026-01 -view-only

# 仅生成打印版
$ ledger generate -month 2026-01 -print-only

# 仅生成 HTML 版
$ ledger generate -month 2026-01 -html-only
```

## Risks / Trade-offs

### 风险 1：Excel 列宽超限
**风险**：打印版 Excel 38 列可能放不下 A4 纸张
**缓解措施**：
- 缩小字号（8-9pt）
- 使用 A3 纸张打印
- 或只对余额列分栏（降级方案）

### 风险 2：HTML 浏览器兼容性
**风险**：不同浏览器打印效果可能有差异
**缓解措施**：
- 测试主流浏览器（Chrome、Firefox、Edge）
- 提供打印操作指引
- 使用 CSS 标准属性，避免浏览器特定前缀

### 风险 3：正反面对齐困难
**风险**：物理打印时左右页可能不对齐
**缓解措施**：
- 提供打印操作指引
- 首次做样板测试，确认对齐效果
- 建议使用 A3 纸张单页打印（如果硬件支持）

### 风险 4：三套输出数据不一致
**风险**：打印版数据错误
**缓解措施**：
- 共用数据源（`entries` 和 `initials`）
- 单元测试覆盖 `centsToDigits` 函数
- 验证策略：只需验证查看版，即可确认打印版正确性

### 风险 5：开发周期超预期
**风险**：可能需要 10+ 天完成所有功能
**缓解措施**：
- 优先实现核心功能（共享函数、打印版 Excel）
- 样式后续迭代，先实现基本功能
- 分阶段交付，每阶段可独立使用

## 实施计划

### 第一阶段：基础建设（1-2 天）
1. 新增 `generator/amount.go`，实现 `centsToDigits` 函数
2. 编写单元测试（边界值：0、9999999999、-1）
3. 抽象列定义结构体
4. 提取共享样式函数

### 第二阶段：打印版 Excel（2-3 天）
1. 实现总分类账打印版
2. 实现多科目明细账打印版
3. 设置打印参数（横向、FitToWidth、重复表头）
4. 测试打印效果

### 第三阶段：HTML 版（2-3 天）
1. 设计 HTML 模板（左右页布局）
2. 编写 CSS 打印样式（边框、字体、页面设置）
3. 实现 Go 模板渲染函数
4. 测试浏览器打印效果

### 第四阶段：集成与测试（1-2 天）
1. 修改 CLI 命令，支持 `-view-only`、`-print-only`、`-html-only`
2. 端到端测试（3 个月凭证）
3. 样板打印测试，确认对齐效果
4. 文档更新

**总计：6-9 天**

---

## 验证策略

### 查看版验证（覆盖约 90%）
- 所有数值正确性
- 余额计算正确性
- 分页逻辑正确性（过次页/承前页数据连续）
- 打印标记正确性（哪些页需要打印）

### 打印版专项验证（剩余 10%）
- 列宽适配（是否超纸边距）
- 行高一致性
- 标题行在两页上的呈现
- 首次打印时做一次完整的对齐测试

### 验证原则
- 数据层面验证查看版，布局层面验证打印版一次
- 首次确认布局正确后，后续每月只需验证查看版
- 因为布局规则是固定的，只要数据正确，打印版必然正确

## 样式探索工作流

### 开发阶段
1. 修改 CSS 文件（在 Go 代码中）
2. 重新编译程序
3. 运行 CLI 命令生成 HTML
4. 用浏览器打开 HTML 文件查看效果
5. 确认样式后，更新 Go 的 `TableStyles`
6. 重新生成 Excel 版本

### 生产阶段
- 格式稳定，维护成本不再是主要问题
- 三个版本的样式保持一致
