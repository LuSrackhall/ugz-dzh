## Design Summary

多科目明细账明细列存在"金额错列"根因：每月 `AppendMLEntries` 和 `WriteMLMonthClosings` 各自按字母序重新排序明细科目并重建 `detailIdx`，导致第2行标题被更新为新顺序，但历史数据行保持原有列位不变——新旧列位脱节。

解决方案：**累积稳定列** — 一旦明细科目被分配到某列，列位永不移动；新增科目仅追加到右侧空列。

## Alternatives Considered

### 方案 A：每月全量重写历史数据行
- **做法**：每次写入前遍历所有历史行，按当月 `detailIdx` 重新排列 H-U 列的值
- **优点**：逻辑简单，不改动列分配策略
- **缺点**：O(n×m) 复杂度，每月都需扫描全量数据；重写历史行会丢失原始写入顺序，且与 excelize 的行级操作模型不匹配
- **为何未采用**：性能差且风险高，不符合累积型工作簿的模型

### 方案 B：预分配固定列（首月锁定）
- **做法**：首月生成时按字母序一次性分配所有可能出现的明细列，后续月份只填充不新增
- **优点**：列位永不变化
- **缺点**：首月无法预见未来科目；需人工预填配置，灵活性差
- **为何未采用**：实际业务中新科目可能随时出现，无法预知

### 方案 C：累积稳定列 + 可选 JSON 配置（采用方案）
- **做法**：读头保序 — 从第2行读取现有列→科目映射，新科目追加到右侧空列；可选 `detailOrder` 配置强制列顺序
- **优点**：无需重写历史数据；支持用户自定义列序；支持空字符串跳列；新科目自动追加并回写配置
- **缺点**：需新增读头逻辑和配置解析
- **为何未采用**：采用此方案

## Agreed Approach

采用方案 C（累积稳定列 + JSON 配置）。核心机制：

1. **读头保序**：`ensureMLSheet` 发现 Sheet 已存在时，从第2行 H-U 读取现有标题，构建 `detailName → columnIndex` 映射。此映射不可变。
2. **只增不调**：新明细科目按 `detailOrder` 配置顺序（若存在）或字母序追加到右侧第一个空列。已存在列绝不移动。
3. **detailOrder 配置**：用户可在 JSON 配置文件中指定 `detailOrder`，包含空字符串跳列支持。配置与科目树解耦——即使科目树中尚无该科目，列标题仍展示配置名称。
4. **自动回写**：新科目追加后自动增量写入 `detailOrder` 配置（用户用 git 管理 JSON，有兜底）。
5. **冲突检测**：如果现有 xlsx 标题与 `detailOrder` 配置不一致 → 报错，提示使用 `-f` 从首月强制重新生成。
6. **统一映射来源**：`AppendMLEntries` 和 `WriteMLMonthClosings` 统一从 Sheet 标题读取 `detailIdx`，不再各自独立排序构建。
7. **超14列报错**：清晰提示已占用列位和可用列数。

## Key Decisions

- `detailOrder` 与科目树完全解耦：配置中预分配的科目即使尚未发生业务，标题也会展示
- 空字符串 `""` 表示跳列占位，新科目不得占用
- 自动回写到 JSON 配置文件（用户用 git 管理作为兜底）
- 明细科目消失后列位保留不回收
